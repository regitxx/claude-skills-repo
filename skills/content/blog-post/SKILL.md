<when_to_use>
Use this skill when:
- Writing technical blog articles
- Sharing engineering lessons learned
- Explaining new features or technologies
- Building thought leadership content
- Creating educational content for developers
- Announcing products or major updates
</when_to_use>

<instructions>
1. **Plan Your Article**
   - Define core message (one takeaway)
   - Identify target reader (experience level)
   - Set appropriate tone (educational, entertaining, authoritative)
   - Create outline with 4-6 sections
   - Plan code examples and diagrams

2. **Hook Your Reader**
   - Start with relatable problem or story
   - Show relevance (why they should care)
   - Promise what they'll learn
   - Keep introduction under 3 paragraphs

3. **Develop Main Content**
   - Each section solves one problem
   - Lead with main idea
   - Support with examples and evidence
   - Use code snippets strategically
   - Explain why, not just how

4. **Sustain Engagement**
   - Use active voice
   - Vary sentence length
   - Include personal experiences when relevant
   - Use analogies for complex ideas
   - Break up text with visuals

5. **Conclude Effectively**
   - Summarize key learnings
   - Provide actionable next steps
   - Invite engagement (comments, questions)
   - Link to related content
   - Keep conclusion concise
</instructions>

<examples>
EXAMPLE 1: Feature Announcement
HEADLINE: "Speeding Up API Calls: Introducing Smart Caching"
STRUCTURE: Problem → Solution → Benefits → Usage → Real-world impact

EXAMPLE 2: Technical Deep Dive
HEADLINE: "Why We Rewrote Our Parser in Rust"
STRUCTURE: Problem → Why Rust → How we did it → Results → Lessons learned

EXAMPLE 3: Educational Tutorial
HEADLINE: "Building a Real-Time Dashboard: A Complete Guide"
STRUCTURE: Overview → Architecture → Step-by-step → Optimization → Deployment
</examples>

<guidelines>
**Do:**
- Write conversational but professional tone
- Lead with the main idea
- Use concrete examples and data
- Include code that actually works
- Assume intermediate technical knowledge
- Engage readers with questions
- Show your personality (authenticity)
- Provide value (not just promotion)

**Don't:**
- Bury the lead (main point should come early)
- Write promotional fluff (no marketing speak)
- Assume deep expertise (explain jargon)
- Use code without explanation
- Make unsupported claims (back up statements)
- Write articles longer than 2000 words
- Focus on your company over the problem
- Ignore comments and feedback

**Code Quality:**
- Examples are complete and tested
- Good error handling demonstrated
- Comments explain the why
- Include expected output
- Show both simple and advanced patterns
</guidelines>
